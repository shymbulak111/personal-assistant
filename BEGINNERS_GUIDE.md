## Что вообще мы сделали?

Мы сделали **приложение «Персональный Ассистент»**. Это как Google Keep + Todoist + ChatGPT, только своё собственное.

Ты открываешь браузер, заходишь на сайт, и можешь:
- Писать задачи (что нужно сделать)
- Писать заметки
- Ставить напоминания
- Общаться с ИИ-ассистентом, который создаёт задачи/напоминания по твоим словам

## Часть 1: Что такое бэкенд и фронтенд?

Представь ресторан.

**Фронтенд** — это зал, где сидят гости. Красиво, удобно, видно меню. Это то, что видит пользователь в браузере. В нашем проекте это файл `index.html`.

**Бэкенд** — это кухня. Гости её не видят, но именно там готовится еда. В нашем проекте это Java-код, который обрабатывает все запросы.

Когда ты нажимаешь кнопку «Создать задачу» — браузер (фронтенд) посылает запрос на кухню (бэкенд), кухня сохраняет задачу в базу данных и отвечает «ок, сохранил».

## Часть 2: Java и Spring Boot — что это?

**Java** — язык программирования. На нём написан бэкенд нашего приложения. Как русский язык, только для компьютеров.

**Spring Boot** — это готовый набор инструментов для Java. Представь, что ты едешь на велосипеде (чистая Java) или на мотоцикле (Spring Boot). Мотоцикл уже с мотором, фарой, тормозами — тебе не нужно их изобретать.

Spring Boot сам:
- Запускает веб-сервер (слушает запросы из браузера)
- Подключается к базе данных
- Управляет безопасностью
- И много другого

Наш код запускается командой `mvn spring-boot:run` — и сразу работает веб-сайт на `http://localhost:8080`.

## Часть 3: База данных PostgreSQL — зачем она?

Данные нужно где-то хранить. Если бы мы хранили всё в памяти компьютера — при перезапуске всё бы терялось.

**PostgreSQL** — это база данных. Представь огромную Excel-таблицу, только очень быстрая и надёжная.

У нас есть такие таблицы (называются «entities» или сущности):

| Таблица | Что хранит |
|---|---|
| `users` | Пользователи: email, пароль, имя |
| `tasks` | Задачи: название, статус, приоритет |
| `notes` | Заметки: заголовок, текст |
| `reminders` | Напоминания: текст, время |
| `chat_sessions` | Сессии чата с ИИ |
| `chat_messages` | Сообщения в чатах |
| `audit_logs` | Лог всех действий |

Связи между таблицами: один пользователь → много задач, много заметок, много напоминаний. Как у одного человека может быть много записей в ежедневнике.

## Часть 4: API — что это такое?

**API** (Application Programming Interface) — это меню ресторана с кнопками.

Ты говоришь: «Хочу задачу», — и даёшь параметры (название, приоритет). Кухня (бэкенд) принимает заказ, готовит (сохраняет в БД), и возвращает результат.

Каждый пункт меню — это **эндпоинт**. У нас они такие:

```
POST   /api/v1/auth/register     — зарегистрироваться
POST   /api/v1/auth/login        — войти в аккаунт

GET    /api/v1/tasks             — получить все задачи
POST   /api/v1/tasks             — создать задачу
PUT    /api/v1/tasks/5           — обновить задачу №5
DELETE /api/v1/tasks/5           — удалить задачу №5
PATCH  /api/v1/tasks/5/complete  — отметить задачу №5 выполненной

(аналогично для заметок и напоминаний)

POST   /api/v1/chat              — написать ИИ-ассистенту
GET    /api/v1/stats             — получить статистику
```

**GET** — получить данные (как спросить «что у вас есть?»)  
**POST** — создать что-то новое  
**PUT** — обновить существующее  
**DELETE** — удалить  
**PATCH** — частично обновить (например, только статус)

Браузер общается с бэкендом в формате **JSON**. Это просто текст с данными:
```json
{
  "title": "Купить молоко",
  "priority": "HIGH",
  "status": "TODO"
}
```

## Часть 5: Авторизация и JWT — как нас узнают?

Представь, что ты пришёл в клуб. На входе тебе дали браслет. Теперь на каждую аттракцию ты показываешь браслет — и тебя пускают без повторной проверки паспорта.

**JWT (JSON Web Token)** — это такой браслет, только цифровой.

1. Ты вводишь email + пароль
2. Сервер проверяет — правильно? Да → выдаёт токен (длинная строка символов)
3. Токен сохраняется в браузере (localStorage)
4. При каждом запросе к API токен отправляется в заголовке: `Authorization: Bearer eyJhbGc...`
5. Сервер видит токен, понимает кто ты, и отдаёт только ТВОИ данные

Токен живёт 24 часа. Потом нужно войти снова (или использовать refresh token на 7 дней).

**Пароль хранится не открытым текстом.** Используется алгоритм **BCrypt** — он превращает «qwerty123» в что-то вроде `$2a$10$XtW3l...`. Это называется хэширование. Даже если взломают базу — пароли не узнают.

## Часть 6: RBAC и ABAC — кто что может делать?

### RBAC (Role-Based Access Control) — контроль по ролям

Как в офисе: сотрудник, менеджер, директор. У каждого свои права.

У нас три роли:
- **ROLE_USER** — обычный пользователь. Может делать всё для себя. Лимит: 50 AI-запросов в месяц.
- **ROLE_PREMIUM** — платный пользователь. Больший лимит AI.
- **ROLE_ADMIN** — администратор. Может всё + доступ к `/api/v1/admin/**`.

### ABAC (Attribute-Based Access Control) — контроль по атрибутам

RBAC говорит «ты пользователь — можешь управлять задачами». Но он не говорит «ты можешь управлять ТОЛЬКО СВОИМИ задачами».

ABAC добавляет это ограничение. Прежде чем удалить задачу №7, система проверяет: «А эта задача №7 вообще ТВОЯ?»

```java
@PreAuthorize("@ownerCheck.isTaskOwner(#id, authentication)")
public void deleteTask(Long id) { ... }
```

Если ты пытаешься удалить чужую задачу — получишь ошибку 403 (Запрещено).

## Часть 7: Redis — зачем ещё одна база?

**Redis** — это очень быстрая база данных, которая хранит данные в оперативной памяти (RAM), а не на диске.

Представь: ты каждый раз идёшь в библиотеку за одной и той же книгой. Утомительно. А если положить книгу на стол рядом — достанешь за секунду.

Redis — это «стол рядом».

Мы используем Redis для:
- **Кэширования** — популярные данные хранятся в Redis, не надо каждый раз лезть в PostgreSQL
- **Rate limiting** — считаем сколько AI-запросов сделал пользователь в этом месяце
- **Сессии** — быстрый доступ к данным авторизации

## Часть 8: Apache Kafka — зачем очередь сообщений?

**Apache Kafka** — это система очередей. Как почтовый ящик для сообщений между частями системы.

Зачем? Представь: пользователь удалил задачу. Нам нужно:
1. Удалить из PostgreSQL
2. Записать в лог
3. Возможно уведомить другие сервисы

Если делать всё последовательно — пользователь ждёт. Если второй шаг упадёт — пользователь получит ошибку, хотя задача уже удалена.

С Kafka: задача удаляется → в Kafka кидается событие «задача удалена» → другие части системы подхватывают событие АСИНХРОННО (в фоне).

Пользователь не ждёт. Если логгирование упадёт — это не его проблема.

У нас Kafka используется для **audit-events** — лога всех действий пользователей.

## Часть 9: Шифрование заметок (AES-256-GCM)

Текст заметок хранится в базе данных **зашифрованным**. Даже если кто-то получит доступ к базе — он увидит только абракадабру.

**AES-256-GCM** — алгоритм шифрования. Очень надёжный, используется в банках и военных системах.

Как это работает:
1. Ты пишешь заметку: «Пароль от сейфа: 1234»
2. Перед сохранением в БД текст шифруется секретным ключом
3. В базе хранится: `a3f9bc1d...` (зашифрованный текст)
4. Когда ты открываешь заметку — текст расшифровывается обратно
5. Всё это происходит автоматически, ты этого не замечаешь

Ключ шифрования хранится в переменной окружения `ENCRYPTION_KEY` — не в коде, не в базе.

## Часть 10: ИИ-ассистент — как он работает?

Мы не делали свой ИИ. Мы подключились к готовому через API.

**Groq** — сервис, который предоставляет доступ к мощным языковым моделям (LLM). Наш ассистент отправляет туда запрос и получает ответ.

### Function Calling — как ИИ создаёт задачи?

Обычный чат просто пишет текст. Наш ИИ умеет **вызывать функции** — то есть делать реальные действия.

Ты пишешь: *«Завтра в 8 утра напомни мне про встречу»*

Вот что происходит:
1. Твоё сообщение + список доступных функций отправляется в LLM
2. LLM решает: «Это запрос на создание напоминания»
3. LLM возвращает: «Вызови функцию `create_reminder` с параметрами title=Встреча, time=08:00 завтра»
4. Наш код выполняет эту функцию — создаёт реальное напоминание в базе данных
5. Пользователь видит: «Напоминание создано!»

### Контекст

Перед каждым запросом к ИИ мы передаём ему контекст — что у тебя есть:
- До 10 активных задач
- До 5 ближайших напоминаний
- До 10 последних заметок

Поэтому ты можешь спросить «что мне сделать сегодня?» и ИИ ответит основываясь на ТВОИХ реальных задачах.

## Часть 11: Тесты — зачем они?

Представь: ты изменил код входа. Как проверить что всё работает? Вручную заходить каждый раз?

**Тесты** — это автоматические проверки. Запускаешь командой `mvn test` и через минуту знаешь: работает или нет.

У нас несколько видов тестов:

**Unit-тесты** — тестируют маленький кусочек кода изолированно. Как проверить одну шестерёнку без всего механизма.

**Интеграционные тесты** — тестируют несколько компонентов вместе. Как запустить весь механизм и посмотреть что получается.

**Testcontainers** — магия! Тесты сами запускают настоящий PostgreSQL, Redis, Kafka в Docker-контейнерах, проводят тесты, и сами же останавливают. Не нужно ничего настраивать вручную.

## Часть 12: Docker — что это и зачем?

Классическая проблема: «У меня работает, а у тебя нет». Разные версии Java, разные настройки ОС...

**Docker** решает это. Он упаковывает приложение со ВСЕМИ его зависимостями в **контейнер** — изолированную коробочку.

Представь: вместо того чтобы везти рецепт и надеяться что у друга есть все ингредиенты — ты везёшь готовое блюдо в контейнере. Всегда одинаково.

**`Dockerfile`** — инструкция как собрать контейнер с нашим приложением:
```
1. Возьми Java 17
2. Скопируй наш код
3. Собери JAR-файл
4. Запусти его
```

**`docker-compose.yml`** — инструкция как запустить ВЕСЬ стек сразу:
- PostgreSQL (база данных)
- Redis (кэш)
- Zookeeper + Kafka (очереди)
- Наше приложение

Одна команда `docker-compose up -d` — и всё работает.

## Часть 13: Kubernetes — зачем это если есть Docker?

Docker запускает контейнеры на ОДНОЙ машине. 

**Kubernetes** управляет контейнерами на МНОГИХ машинах.

Представь: Docker — это один повар. Kubernetes — это ресторанная сеть с управляющим.

Kubernetes умеет:
- Запустить 2 (или 10) копии приложения для нагрузки
- Если одна копия упала — автоматически поднять новую
- Распределить нагрузку между копиями

У нас есть файлы в папке `k8s/`:
- `deployment.yml` — запустить 2 копии приложения
- `service.yml` — как к ним обращаться
- `configmap.yml` — настройки (хост БД, порты)

## Часть 14: GitHub Actions (CI/CD) — автоматизация

**CI/CD** расшифровывается как Continuous Integration / Continuous Deployment.

Простыми словами: каждый раз когда ты заливаешь код на GitHub, автоматически запускается:
1. Компиляция кода (проверка нет ли ошибок)
2. Запуск всех тестов
3. Сборка Docker-образа
4. Деплой на сервер

Это как конвейер на заводе. Ты кидаешь сырьё (код) на вход — на выходе готовый продукт (работающее приложение на сервере).

Наш пайплайн описан в `.github/workflows/ci.yml`.

## Часть 15: GDPR — что это?

**GDPR** — европейский закон о защите персональных данных. Если коротко: у пользователя есть права на свои данные.

Мы реализовали два права:

**Право на экспорт данных:** `GET /api/v1/users/me/export`  
Ты можешь скачать все свои данные в JSON-файле: задачи, заметки, напоминания, история чатов.

**Право на удаление аккаунта:** `DELETE /api/v1/users/me`  
Полное удаление всего что связано с тобой из базы данных. Каскадное — удаляется пользователь, и автоматически всё остальное.

## Часть 16: Фронтенд — как сделан сайт?

Весь интерфейс — это один файл `index.html`. 

Без React, без Vue, без Angular. Просто:
- **HTML** — структура страницы (как скелет)
- **CSS** — внешний вид (как одежда)
- **JavaScript** — логика (как мозг)

**Темы:** При нажатии ☀️/🌙 меняется тема. Реализовано через CSS-переменные:
```css
:root { --bg: #0a0a0f; --text: #e8e6f0; }  /* тёмная */
[data-theme="light"] { --bg: #f5f5f7; --text: #1a1a2e; }  /* светлая */
```

**Языки (EN/RU):** Весь текст в JS-объекте. При переключении языка все надписи обновляются.

**Авторизация:** JWT токен хранится в `localStorage` браузера. При каждом запросе добавляется в заголовок:
```javascript
headers: { 'Authorization': 'Bearer ' + localStorage.getItem('token') }
```

## Часть 17: Архитектура кода (Clean Architecture)

Код разделён на слои, как торт:

```
┌─────────────────────────────────┐
│  Controller (HTTP слой)         │  ← принимает запросы из браузера
├─────────────────────────────────┤
│  Service (бизнес-логика)        │  ← думает что делать
├─────────────────────────────────┤
│  Repository (доступ к данным)   │  ← работает с базой данных
├─────────────────────────────────┤
│  Domain / Entity (модели)       │  ← описывает объекты (Task, Note...)
└─────────────────────────────────┘
```

**Зачем так?** Если нужно поменять базу данных с PostgreSQL на MySQL — меняешь только Repository. Бизнес-логика в Service не трогается. Это называется «разделение ответственностей».

**DTO (Data Transfer Object)** — отдельные объекты для передачи данных. Чтобы не отдавать наружу всё подряд из базы данных (например, хэш пароля).

**Mapper** — автоматически конвертирует Entity (объект из БД) в DTO и обратно. Мы используем MapStruct — он генерирует код конвертации при компиляции.

## Часть 19: Как это всё запустить самому?

### Шаг 1 — Что нужно установить
- **Java 17** — язык программирования
- **Maven** — сборщик Java-проектов (как npm для Node.js)
- **Docker Desktop** — для контейнеров

### Шаг 2 — Клонировать код
```bash
git clone https://github.com/shymbulak111/personal-assistant.git
cd personal-assistant
```

### Шаг 3 — Запустить инфраструктуру
```bash
docker-compose up -d
```
Это запустит PostgreSQL, Redis, Kafka в фоне.

### Шаг 4 — Запустить приложение
```bash
mvn spring-boot:run
```

### Шаг 5 — Открыть в браузере
```
http://localhost:8080
```

Зарегистрируйся → войди → пользуйся!
